## Charter Purpose

This charter defines the governing doctrine, operating model, and institutional architecture for agentic loops within the Florence ecosystem.

Within Florence-X, agentic loops are not treated as scripts, assistants, or isolated automations. They are treated as governed operational systems embedded inside human institutions. Their legitimacy depends on orchestration, stewardship, auditability, and ethical control rather than model capability alone.

This charter is intended to function simultaneously as:

- a doctrinal foundation
- an operating charter
- a governance standard
- a systems blueprint

for Florence-X, NAIO, EDENA, NIN, and NINE²-aligned AI workforce systems.

## I. Foundational Doctrine

### 1. Governance Before Autonomy

No agent, workflow, or loop may operate in production without governance controls in place first.

At minimum, this includes:

- registration
- EDENA classification
- steward assignment
- memory scope definition
- tool permissioning
- audit logging

Autonomy is permitted only inside a governed boundary.

### 2. Human Authority Layer

Humans retain final authority for judgment, ethical interpretation, regulated actions, and irreversible decisions.

AI may execute defined workflows, but humans remain responsible for oversight, authorization, and accountability.

### 3. Orchestration Over Intelligence

The most powerful AI system is not the one with the most capable isolated model.

It is the one with the best orchestration architecture: a governed control plane coordinating specialized agents, tools, memory systems, and human stewards.

### 4. Agents as Operational Actors

AI agents are not mere software utilities. They are operational actors inside institutional systems.

Therefore every agent must have:

- an identity
- a defined role
- a governance scope
- a risk tier
- a lifecycle
- a human steward

### 5. Modular Capability as the Basis of Scale

AI workforces scale through modular skills, not monolithic assistants.

CoworkOS skills provide the reusable capability layer that allows agents to invoke standardized functions through governed interfaces.

### 6. Sovereign Infrastructure as a Governance Requirement

Agentic systems must run on controlled infrastructure with isolation, replaceability, and observability.

Containerization, segmented memory, controlled execution environments, and auditable system boundaries are governance requirements, not optional technical preferences.

### 7. Continuous Improvement Through Feedback

Every loop must include measurement, review, learning, and system update.

A Florence loop is not complete when it executes. It is complete when it improves the system that produced it.

## III. Definition of an Agentic Loop

An agentic loop is a governed workflow cycle in which agents, skills, tools, memory systems, and human stewards repeatedly transform signals into action and action into improved institutional intelligence.

### Standard Loop Pattern

```text
Signal Detection
→ Analysis
→ Decision Proposal
→ EDENA Classification
→ Action or Escalation
→ Measurement
→ Learning
→ System Update
```

A loop without governance gates, monitoring, and feedback is not an agentic loop under this charter. It is merely an automation.

- tool access verification

## VII. EDENA Risk Doctrine

EDENA is the ethical decision engine embedded in Florence operations.

Its core rule is simple:

**All AI actions must be classified before execution.**

### Risk Zones

| Zone | Meaning | Automation Rule | Human Oversight |
|---|---|---|---|
| Green | Low-risk mechanical or informational tasks | Fully automated | Periodic review |
| Yellow | Operational actions affecting systems or stakeholders | Conditional automation | Approval queue or threshold review |
| Red | High-risk, irreversible, clinical, legal, financial, or infrastructure actions | Blocked by default | Mandatory authorization |

### Escalation Path

```text
Level 1 — Automated Monitoring
Level 2 — Steward Review
Level 3 — Governance Escalation
Level 4 — System Intervention
```

No Red-tier decision may be executed autonomously.

## VIII. Memory and Data Governance

Florence-X enforces memory zoning to prevent unsafe data access and cross-contamination.

### Memory Zones

| Zone | Description |
|---|---|
| Green | Public or low-risk knowledge |
| Yellow | Internal operational data |
| Red | Sensitive or regulated information |

Agents may access only those memory zones explicitly permitted by their EDENA classification and registry configuration.

Cross-zone transfer requires governance approval.

### PHI Safeguards

Because the Florence ecosystem is healthcare-oriented and nurse-led, protected data governance is mandatory.

Systems handling PHI or regulated information must enforce:

- encryption at rest
- encrypted transmission
- role-based access control
- access logging
- restricted storage boundaries
- prohibition on model training use
- prohibition on transmission to unauthorized external tools

If PHI exposure risk is detected, the workflow must be suspended, treated as a Red incident, and escalated immediately.

## IX. Auditability and Control Requirements

Every loop execution must produce a verifiable audit record.

At minimum, audit logs should capture:

```text
timestamp
agent identifier
action description
edena classification
data sources
tool usage
approval status
steward decision if applicable
resolution rationale for escalations
```

Logs must be immutable, retained according to policy, and available for operational, governance, security, and compliance review.

| Metric | Purpose |
|---|---|
| task success rate | reliability |
| error rate | safety |
| latency | operational performance |
| compute or token cost | efficiency |
| escalation rate | governance effectiveness |
| anomaly frequency | safety monitoring |
| approval turnaround time | oversight efficiency |

### Incident Response Pattern

```text
Detect
→ Contain
→ Investigate
→ Remediate
→ Review
→ Update policy, prompts, skills, or routing logic
```

Agents may be suspended, restricted, or retired if they fail governance, reliability, or safety requirements.

## XIV. Deployment Model

A Florence-aligned organization should deploy agentic loops in phases.

### Phase 1 — Establish Governance and Control Plane

Stand up the governance body, define policies, configure Florence-X control plane components, and establish memory, tool, and logging boundaries.

### Phase 2 — Build the Initial Workforce and Skills Library

Launch a bounded initial workforce and implement core skills such as research synthesis, briefing generation, compliance checking, workflow execution, and dashboard creation.

### Phase 3 — Activate Scheduled Loops

Implement recurring scans, reviews, and briefings through governed schedules.

### Phase 4 — Instrument Monitoring and Escalation

Deploy dashboards, anomaly detection, incident handling, and governance review cadence.

### Phase 5 — Expand into Education, Community, Media, and Venture Loops

Once the core loop infrastructure is governed and stable, expand into the broader Florence flywheel.
